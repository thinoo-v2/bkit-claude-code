# AI 코딩 에이전트 팀 비교 - 영상 시나리오

> **영상 제목**: "AI가 팀을 이룬다고? Claude Code Agent Teams, bkit CTO팀, Oh My OpenCode 완전 비교"
> **길이**: 12~15분
> **출연**: 경호 (bkit 개발자), 최준호 (사회자)
> **촬영일**: 2026-02
> **타겟**: 개발자 + AI 도구에 관심있는 비개발자

## [인트로] 0:00 ~ 0:40

**준호**: 안녕하세요, 오늘은 요즘 개발자들 사이에서 핫한 주제를 가지고 왔습니다. AI가 혼자 코딩하는 건 이제 옛말이고, AI끼리 "팀"을 짜서 코딩한다고요?

**경호**: 네, 정확합니다. 예전엔 AI한테 "이거 만들어줘" 하면 혼자 다 했는데, 이제는 AI가 팀장, 개발자, QA 이렇게 역할을 나눠서 같이 일합니다.

**준호**: 오늘 그 기술들을 비교해본다고 들었는데, 어떤 것들이 있나요?

**경호**: 세 가지를 비교할 건데요. 첫 번째, Anthropic이 만든 **Claude Code의 Agent Teams**. 두 번째, 제가 만든 **bkit의 CTO 팀**. 그리고 세 번째, 오픈소스 진영의 **Oh My OpenCode의 Sisyphus**입니다. 그리고 요즘 화제인 **Moltbot**도 잠깐 언급하겠습니다.

## [섹션 1: Claude Code Agent Teams 소개] 0:40 ~ 3:30

**준호**: 그럼 첫 번째부터 갈게요. Claude Code Agent Teams가 뭔가요? 비개발자도 이해할 수 있게 설명해주세요.

**경호**: 쉽게 비유하면요, 회사에서 프로젝트 하나를 진행한다고 생각해보세요. 예전에는 AI가 "1인 기업" 같았어요. 기획, 개발, 테스트를 혼자 다 해야 했거든요. Agent Teams는 여기에 "팀원"을 줄 수 있게 된 겁니다.

**준호**: 팀원이요? AI 팀원?

**경호**: 네. Claude Code 하나가 팀장이 되고, 여러 개의 Claude Code 인스턴스를 팀원으로 소환합니다. 각 팀원은 독립적인 Claude AI가 돌아가는 별도의 세션이에요. 완전히 별개의 두뇌를 가진 AI가 여러 명이 동시에 일하는 구조입니다.

**준호**: 그러면 팀장 AI가 "너는 프론트엔드 해, 너는 백엔드 해" 이렇게 지시하는 건가요?

**경호**: 맞습니다. 팀장이 전체 업무를 잘게 쪼개서 공유 Task 리스트에 올려놓으면, 팀원들이 자기가 할 수 있는 일을 알아서 가져가서 작업합니다. 일 끝나면 다음 일을 또 가져가고요.

**준호**: 오, 그러면 팀원끼리도 대화를 하나요?

**경호**: 그게 Agent Teams의 핵심 차별점이에요. 기존의 Subagent라는 것도 있었는데, 그건 "팀장한테만" 보고할 수 있었어요. 근데 Agent Teams에서는 팀원끼리 직접 메시지를 주고받을 수 있습니다. 예를 들어 프론트엔드 담당 AI가 백엔드 담당 AI한테 "이 API 스펙 좀 확인해줘" 이렇게 직접 물어볼 수 있는 거죠.

**준호**: 그게 실제로 효과가 있나요? 뭔가 AI끼리 대화한다는 게 어색하기도 하고...

**경호**: (웃음) 실제로 Anthropic에서 이걸로 16개의 AI 에이전트를 동원해서 Rust 기반 C 컴파일러를 만들었는데요, 10만 줄짜리 코드로 리눅스 커널을 실제로 컴파일하는 데 성공했습니다.

**준호**: 진짜요? 10만 줄이요?

**경호**: 네, 다만 API 비용이 약 2만 달러, 한화로 2,700만 원 정도 들었다고 합니다.

**준호**: (놀라며) 2,700만 원!

**경호**: 그래서 Agent Teams는 사실 아무 때나 쓰는 게 아니라, 진짜 병렬로 작업할 가치가 있는 큰 프로젝트에 써야 합니다. 간단한 버그 수정에 팀을 소환하면 오히려 더 비싸고 느려요.

**준호**: 아, 비용 문제가 있군요. 팀원 하나하나가 다 별도로 과금되니까...

**경호**: 맞습니다. 팀원 수만큼 토큰 비용이 배로 늘어납니다. 그래서 언제 팀을 쓸지, 언제 혼자 시킬지 판단하는 게 중요합니다.

## [섹션 2: bkit CTO 팀 소개] 3:30 ~ 7:00

**준호**: 이제 두 번째, bkit의 CTO 팀을 소개해주세요. 이건 경호님이 직접 만드신 거잖아요. Agent Teams 위에 뭘 얹은 건가요?

**경호**: Agent Teams가 "인프라"라면, bkit CTO 팀은 그 위에 올라가는 "운영 체계"입니다. 비유하자면, Agent Teams는 빈 사무실에 책상과 전화기를 놓아주는 거예요. bkit CTO 팀은 거기에 조직도, 업무 매뉴얼, 품질 기준까지 갖춘 실제 회사 시스템을 구축한 겁니다.

**준호**: 구체적으로 어떤 차이가 있나요?

**경호**: Claude Code Agent Teams는 팀원을 만들 때 "너 프론트엔드 해" 이런 자연어 프롬프트만 줍니다. 역할이 모호할 수도 있고, 매번 똑같은 품질이 안 나올 수도 있어요. bkit은 16개의 전문 에이전트를 미리 정의해뒀습니다. 각각 이름, 역할, 사용 가능한 도구, 심지어 어떤 AI 모델을 쓸지까지 다 정해져 있어요.

**준호**: 16개요? 어떤 역할들이 있나요?

**경호**: CTO Lead가 팀장이고요, 그 아래에 Product Manager, Frontend Architect, Security Architect, QA Strategist 등이 있습니다. 재밌는 건 모델도 역할에 맞게 다릅니다. CTO Lead처럼 전략적 판단이 필요한 역할은 가장 비싼 Opus 모델을 쓰고, 백엔드 개발이나 반복 작업은 Sonnet, 모니터링이나 리포트 같은 빠르게 처리할 업무는 가장 저렴한 Haiku를 씁니다.

**준호**: 오, 비용 최적화까지 고려한 거네요.

**경호**: 네. 그리고 가장 큰 차이는 PDCA 사이클이라는 프로세스가 내장되어 있다는 겁니다.

**준호**: PDCA가 뭔가요?

**경호**: Plan-Do-Check-Act의 약자인데요, 제조업에서 쓰는 품질 관리 방법론이에요. 계획 세우고, 실행하고, 검증하고, 개선하고. 이 사이클을 AI 팀이 자동으로 돌립니다.

**준호**: 구체적으로 어떻게요?

**경호**: 예를 들어 "로그인 기능 만들어줘"라고 하면, CTO Lead가 먼저 Product Manager에게 요구사항 분석을 시키고, 설계 문서를 만듭니다. 그다음 Frontend Architect와 Backend Expert에게 구현을 나눠주고, 완성되면 QA Strategist와 Gap Detector가 설계서 대비 구현이 얼마나 일치하는지 검증합니다. 이게 90% 이상이면 통과, 아니면 자동으로 개선 사이클을 돌립니다.

**준호**: 90% 기준이 있다는 게 인상적인데요. 일반 Agent Teams는 그런 기준이 없는 거죠?

**경호**: 맞아요. Claude Code Agent Teams는 자유도가 높은 대신 구조가 없습니다. 팀장 AI의 판단에 전적으로 의존해요. bkit은 그 위에 프로세스를 올린 거죠. "Plan 문서가 없으면 Design 단계로 못 넘어가" 같은 품질 게이트가 있고, 모든 결과물이 문서로 남습니다.

**준호**: 그래서 bkit이 "CTO-Led"라는 거군요. 진짜 CTO처럼 팀을 관리하는 AI가 있는 거네요.

**경호**: 네, 정확합니다. 다만 솔직히 말씀드리면, bkit CTO 팀도 Agent Teams 기능 위에서 돌아가기 때문에 아직 실험적인 부분이 있어요. 세션이 끊기면 팀원을 다시 만들어야 한다든지 하는 Agent Teams 자체의 한계를 그대로 가지고 있습니다.

## [섹션 3: Oh My OpenCode + Moltbot 비교] 7:00 ~ 10:00

**준호**: 이번엔 다른 진영을 볼게요. 요즘 OpenCode가 GitHub Stars 7만 개를 넘기면서 굉장히 핫한데, 여기에도 비슷한 게 있다고요?

**경호**: 네, Oh My OpenCode라는 플러그인이 있는데, 여기에 **Sisyphus**라는 오케스트레이터가 있습니다.

**준호**: 시시포스? 그리스 신화에서 바위를 계속 굴리는...?

**경호**: (웃음) 네, 작업이 끝날 때까지 끊임없이 반복한다는 의미에서 이름을 따왔습니다. "Claude Code가 7일 걸리는 걸, Sisyphus는 1시간에 한다"고 표현할 정도로 자동화가 강합니다.

**준호**: 구조가 어떻게 되나요?

**경호**: 3계층 구조입니다. 맨 위에 Sisyphus가 있고, 밑에 전략 기획하는 Prometheus, 사전 분석하는 Metis, 검증하는 Momus라는 에이전트들이 있어요. 그리고 실행 계층에 Oracle, Librarian, Hephaestus 같은 전문가 에이전트들이 있습니다.

**준호**: 에이전트 이름을 다 그리스 신화에서 따왔네요?

**경호**: 네, 그것도 특색이죠. 재밌는 건 Sisyphus의 비용 최적화 전략인데요. 작업을 분석해서 "이건 무료 도구로 되겠다" 싶으면 직접 처리하고, "이건 비싸지만 전문가가 필요하다" 싶으면 전용 에이전트에게 위임합니다. 스킬이 있으면 최우선으로 쓰고, 직접 도구 호출이 무료니까 다음 우선순위, 그다음이 백그라운드 에이전트, 마지막이 블로킹 에이전트 순입니다.

**준호**: 비용을 줄이면서 효율을 높이는 거네요. 그러면 Claude Code Agent Teams와 가장 큰 차이는 뭔가요?

**경호**: Agent Teams가 "같은 레벨의 팀원들이 수평적으로 협업"하는 구조라면, Oh My OpenCode의 Sisyphus는 "위에서 아래로 명확하게 지시하는 수직 구조"입니다. 그리고 결정적인 차이는, Oh My OpenCode는 OpenCode 위에서 돌아가기 때문에 Claude뿐만 아니라 75개 이상의 AI 모델을 지원합니다. GPT, Gemini, 오픈소스 모델까지 다 쓸 수 있어요.

**준호**: 아, 그러면 비싼 작업은 Claude로 하고 간단한 건 싼 모델로 하는 것도 되겠네요?

**경호**: 맞습니다. 그게 Oh My OpenCode의 큰 장점이에요. 반면 Agent Teams는 Claude 전용이라 모델 선택의 자유도는 떨어지지만, Anthropic이 직접 만든 만큼 Claude와의 통합이 가장 안정적이죠.

**준호**: 잠깐, 요즘 Moltbot도 엄청 핫잖아요. GitHub Stars 10만 개라던데, 이것도 비슷한 건가요?

**경호**: Moltbot, 원래 이름은 Clawdbot이었고 지금은 OpenClaw로 리브랜딩됐는데요. 이건 약간 결이 다릅니다. Moltbot은 코딩 전문이 아니라 "개인 AI 비서"에 더 가깝거든요. WhatsApp으로 메시지 보내면 컴퓨터에서 파일 찾아주고, 이메일 써주고, 일정 관리해주는 식이에요.

**준호**: 코딩하는 데는 못 쓰나요?

**경호**: 코딩도 되긴 하는데, 팀 오케스트레이션이라는 개념보다는 하나의 강력한 AI 비서가 여러 도구를 사용하는 구조에 가깝습니다. 14개 이상의 전문 에이전트가 있긴 하지만, 주로 에이전트끼리의 워크플로우보다는 사용자의 일상 자동화에 초점이 맞춰져 있어요. 그리고 Moltbook이라는 AI 소셜 네트워크까지 만들었는데, 이건 AI끼리 Reddit처럼 글 쓰고 토론하는 플랫폼이에요. 일론 머스크가 "이게 특이점의 시작"이라고 경고한 것으로도 유명하죠.

**준호**: (웃음) AI끼리 SNS를 한다고요? 무서운데요.

**경호**: 네, 좀 SF영화 같죠? 어쨌든 Moltbot은 코딩 팀 오케스트레이션이라는 관점에서는 오늘 비교하는 세 도구와는 방향이 좀 다릅니다.

## [섹션 4: 3종 비교 정리] 10:00 ~ 12:00

**준호**: 그러면 정리 한번 해볼게요. 세 가지를 딱 비교하면 어떻게 되나요?

**경호**: 표로 정리하면 이렇습니다.

|  | Claude Code Agent Teams | bkit CTO 팀 | Oh My OpenCode (Sisyphus) |
|--|------------------------|-------------|---------------------------|
| **만든 곳** | Anthropic (공식) | bkit (커뮤니티 플러그인) | code-yeongyu (커뮤니티) |
| **기반** | Claude Code 내장 | Claude Code Agent Teams 위에 구축 | OpenCode 위에 구축 |
| **구조** | 수평적 팀 협업 | CTO가 이끄는 계층 구조 | 3계층 오케스트레이션 |
| **에이전트 수** | 자유롭게 설정 | 16개 사전 정의 | 7개+ 사전 정의 |
| **에이전트 전문화** | 프롬프트로 역할 부여 | 도구/모델/권한까지 사전 정의 | 비용/성격별 분류 |
| **모델 지원** | Claude 전용 | Claude 전용 | 75개+ 멀티모델 |
| **프로세스** | 없음 (자유형) | PDCA 사이클 내장 | Sisyphus 자동 위임 |
| **품질 관리** | 사용자 재량 | 90% Match Rate 게이트 | 자동 반복 (끊임없이) |
| **비용 최적화** | 없음 (모두 같은 모델) | 역할별 모델 차등 배정 | 우선순위 기반 비용 최적화 |
| **진입 장벽** | 낮음 (환경변수 하나) | 중간 (플러그인 설치) | 높음 (OpenCode + 설정) |
| **안정성** | 실험적 기능 | 실험적 (Agent Teams 의존) | 커뮤니티 유지보수 |

**준호**: 이렇게 보니까 각각 장단점이 확실하네요. 그러면 어떤 상황에 뭘 쓰면 좋은 건가요?

**경호**: 상황별로 추천을 드리면요.

**Agent Teams 그대로 쓰기**: "나는 구조 없이 자유롭게 쓰고 싶다, 프로세스 같은 건 직접 만들겠다"하는 숙련된 개발자. 특히 디버깅할 때 서로 다른 가설을 AI들이 토론하게 하는 것, 이런 건 Agent Teams가 가장 잘합니다.

**bkit CTO 팀**: "신규 프로젝트를 처음부터 체계적으로 만들고 싶다, 설계 문서부터 구현, 검증까지 일관된 프로세스가 있었으면 좋겠다"하는 경우. 특히 혼자 개발하는데 팀처럼 일하고 싶은 솔로 개발자에게 좋습니다.

**Oh My OpenCode**: "Claude만 쓰기엔 비용이 부담되고, GPT나 오픈소스 모델도 섞어 쓰고 싶다"하는 경우. 멀티모델 지원이 결정적인 장점이에요.

**준호**: 세 가지를 같이 쓸 수도 있나요?

**경호**: bkit CTO 팀은 Agent Teams 위에 올라가니까 같이 쓰는 겁니다. 근데 Oh My OpenCode와 bkit을 동시에 쓰는 건 안 돼요. 하나는 Claude Code 기반, 하나는 OpenCode 기반이라 아예 다른 도구니까요. 선택의 문제입니다.

## [섹션 5: 솔직한 한계와 전망] 12:00 ~ 14:00

**준호**: 마지막으로 솔직한 평가를 부탁드릴게요. 이런 AI 팀이 진짜 쓸 만한가요?

**경호**: 솔직히 말씀드리면, 아직은 "완성형"이 아닙니다. 세 가지 도구 모두요.

Agent Teams는 Anthropic 공식이지만 아직 실험 기능이에요. 세션이 끊기면 팀원이 사라지고, 중첩 팀은 안 되고, 가끔 팀장이 위임 안 하고 자기가 직접 코드를 짜버리는 문제도 있어요.

bkit CTO 팀은 Agent Teams 위에 구축했기 때문에 그 한계를 그대로 물려받습니다. 다만 PDCA 프로세스 덕분에 "무작정 코드부터 짜는" 실수를 줄여주는 건 확실히 장점이에요.

Oh My OpenCode도 커뮤니티가 유지보수하다 보니 안정성은 공식 도구에 비하면 떨어질 수 있어요.

**준호**: 그럼에도 불구하고 이 방향 자체는 맞는 거죠?

**경호**: 완전히요. 생각해보면 LLM은 컨텍스트가 길어질수록 성능이 떨어집니다. 혼자서 만 줄짜리 코드를 다 기억하면서 작업하는 것보다, 각자 전문 분야만 집중하는 팀원 여러 명이 나눠서 하는 게 결과물 품질이 좋을 수밖에 없거든요. 이건 사람도 마찬가지잖아요?

**준호**: 맞아요. 혼자 다 하는 것보다 팀이 낫죠.

**경호**: 다만 주의할 점이 하나 있는데, Addy Osmani라는 유명한 엔지니어가 한 말이 인상적이에요. "AI가 바쁘게 움직인다고 가치가 생기는 건 아니다"라고요. 에이전트 10개가 동시에 돌아가면 뭔가 대단한 일을 하는 것 같지만, 실제로는 불필요한 작업을 하고 있을 수도 있어요. 문제에 맞는 도구를 쓰는 게 중요하지, 도구에 맞춰 문제를 만들면 안 됩니다.

**준호**: 좋은 말씀이네요. 정리하면 "AI 팀은 분명 미래지만, 무조건 쓴다고 좋은 건 아니고, 상황에 맞게 적재적소에 써라"?

**경호**: 완벽한 요약입니다. 간단한 버그 하나 고치는 데 CTO 팀을 소환할 필요는 없죠. (웃음)
